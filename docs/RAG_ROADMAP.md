# MindRAG 优化路线图

基于 Harness 工程视角（Prepare → Execute → Verify → Cleanup）的 RAG 系统优化计划。

## Phase 1: Prepare — 数据与索引质量

> 目标：提升入库文档质量，让检索"原材料"更精准

| ID | 优化项 | 当前状态 | 目标方案 | 影响 | 复杂度 |
|----|--------|---------|---------|------|--------|
| P1 | Semantic Chunking | 固定 512 字符分块 | Embedding 相似度切分，断在语义边界 | 检索精确度 ↑30% | 中 |
| P2 | 分层索引 | 单一索引 | 摘要索引 + 详细索引，先粗筛再加载 | 大文档检索速度 ↑50% | 中 |
| P3 | 文档元数据增强 | 仅文件名 | 标题/页码/段落标题/文档类型/创建时间 | 过滤精度 ↑ | 低 |
| P4 | 文档预处理 | 无 | 去水印、去页眉页脚、表格转 Markdown | PDF 解析质量 ↑ | 高 |
| P5 | Parent-Child Chunking | 无层级 | 检索用小 Chunk(256)，返回完整 Parent(1024) | 引用完整性 ↑ | 中 |
| P6 | 增量索引 | 全量重建 | 按文档粒度增量更新，避免全量重新向量化 | 索引效率 ↑ | 中 |

## Phase 2: Execute — 检索与生成增强

> 目标：提升回答质量和检索命中率

| ID | 优化项 | 当前状态 | 目标方案 | 影响 | 复杂度 |
|----|--------|---------|---------|------|--------|
| E1 | Re-Ranking | 无 | ES 粗排 30 条 → BGE-Reranker/Cohere 精排 Top 5 | 精确度 ↑50% | 低 |
| E2 | Query Rewriting | 原始查询 | LLM 改写用户查询为更精确的检索表达 | 召回率 ↑30% | 低 |
| E3 | HyDE | 无 | LLM 先生成假设答案，用答案向量检索 | 召回率 ↑40% | 低 |
| E4 | Query Expansion | 单查询 | 一个查询扩写成 3 个变体，分别检索合并去重 | 召回覆盖率 ↑ | 低 |
| E5 | Multi-Hop Retrieval | 单次检索 | 检索 → 分析 → 补查，多轮迭代 | 复杂查询效果 ↑ | 高 |
| E6 | Self-RAG | 无 | Agent 自我评估检索质量，不够好自动重查 | 端到端质量 ↑ | 高 |

## Phase 3: Verify — 质量闭环

> 目标：建立可度量的质量保障体系

| ID | 优化项 | 当前状态 | 目标方案 | 影响 | 复杂度 |
|----|--------|---------|---------|------|--------|
| V1 | RAGAS 离线评估 | 已实现 | 扩展为定时自动评估 + 趋势告警 | 质量可见性 ↑ | 低 |
| V2 | 在线反馈闭环 | 无 | 用户 👍/👎 → 反馈信号改善检索排序 | 持续优化 ↑ | 中 |
| V3 | 回归测试集 | 无 | 50+ 标准 QA 对，每次改动自动验证 | 回归保护 ↑ | 低 |
| V4 | A/B 测试框架 | 无 | 新旧策略并行运行，对比指标 | 策略验证 ↑ | 高 |
| V5 | 幻觉检测 | 无 | 生成后自动检测答案是否被检索内容支撑 | 可信度 ↑ | 中 |

## Phase 4: Cleanup — 持续维护

> 目标：保持系统长期健康运行

| ID | 优化项 | 当前状态 | 目标方案 | 影响 | 复杂度 |
|----|--------|---------|---------|------|--------|
| C1 | 文档生命周期 | 手动删除 | TTL 自动过期 + 孤儿向量/Chunk 清理 | 存储成本 ↓ | 低 |
| C2 | ES 索引优化 | 默认配置 | 冷热分离、段合并策略、定期 Force Merge | 检索速度 ↑ | 中 |
| C3 | Embedding 漂移检测 | 无 | 监控向量分布变化，模型升级时自动重索引 | 系统稳定性 ↑ | 中 |
| C4 | 脏数据清理 | 无 | 定时扫描 ES vs MySQL vs MinIO 一致性 | 数据一致性 ↑ | 低 |
| C5 | 用量趋势分析 | 无 | Token 消耗/检索延迟/索引量趋势看板 | 容量规划 ↑ | 中 |

## 推荐实施顺序

```
第一轮（低复杂度，高回报）：
  E1 Re-Ranking → E2 Query Rewriting → E3 HyDE → E4 Query Expansion
  预期：检索质量综合提升 60-80%

第二轮（质量闭环）：
  V3 回归测试集 → V1 评估自动化 → V2 在线反馈
  预期：质量可量化，持续优化有依据

第三轮（数据质量）：
  P3 元数据增强 → P1 Semantic Chunking → P2 分层索引
  预期：奠定高性能 RAG 基础

第四轮（高级能力）：
  E6 Self-RAG → E5 Multi-Hop → V4 A/B 测试
  预期：达到企业级 RAG 水准
```

## 技术改造注意

- 全部改造保持 API 兼容，前端无需改动
- 新策略通过 Feature Flag 切换，支持灰度发布
- 每个 Phase 完成后跑 RAGAS 评估对比基线
